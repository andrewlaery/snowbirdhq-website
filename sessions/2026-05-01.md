## Findings

### Architecture: where each kind of content belongs (the long-term model)

After both phases of i18n work and the cd1311e diagnosis, the right model split is by content *shape*, not by location:

| Shape | Belongs in | Loaded by |
|---|---|---|
| Atomic facts (WiFi, beds, check-in) | `data/sot/properties/<slug>/facts.yaml` | `<PropertyQuickInfo>` etc. |
| Cross-property boilerplate (House Rules, Critical Info, Queenstown Essentials) | `data/sot/_shared/{,zh/}*.md` | `<HouseRulesBase>` etc. via `loadShared(name, lang)` |
| Cross-property appliance how-to (Bosch hob, Sonos) | `data/sot/_appliances/{,zh/}<model>.md` | `<ApplianceSet slug=… lang=…>` (filesystem-resolved, no registry) |
| **Per-property narrative** (THIS heating, THIS AV setup, THIS drinking water) | **MDX body** in `content/docs/properties/<slug>/*.mdx` | Fumadocs catch-all renderer |
| Per-locale overlay | `content/docs/zh/properties/<slug>/*.mdx` | Same catch-all (different URL prefix) |
| UI chrome (filter labels, suggestion chips) | `data/sot/_strings/{en,zh}.yaml` | `loadStrings(lang)` |
| Queenstown Insights data | `data/sot/queenstown/{,zh/}insights.yaml` | `loadQueenstownInsights(lang)` |

The mistake in commit `cd1311e` (28 Apr, prior session): tried to push ALL content into structured YAML, including narrative. YAML is the wrong medium for narrative — multi-line strings are brittle, no editor syntax highlighting, painful PR diffs, awkward to translate. MDX bodies are the right home for per-property narrative.

### cd1311e regression — full extent

Audit comparing `cd1311e^` against current MDX revealed ~126 KB of guest-facing content was hollowed out across **11 properties × 2 pages** (`welcome-house-rules` + `critical-info`), in addition to the `user-instructions` regression spotted by the user. PR #11 fixed user-instructions; PR #13 finished the job on the other two pages. Total restored across the day: ~250 KB.

The commit message on `cd1311e` asserted "no content lost" — that assertion was unverified and wrong. Lesson: cut-to-composition migrations need a content-parity check (body-char-count diff or rendered-HTML diff) before merging.

### Best-practice fix vs band-aid (user feedback)

When I initially proposed a hybrid "revert NOW, backfill SOT LATER" path, the user pushed back: "What is the best approach here for long term success, what would we do if we did it based on best practice?" — and "I made some recent updates to 10b-delamare (the heating), don't lose those updates."

Recorded as memory: when the user reports a regression, default to (a) explaining the architectural lesson, (b) preserving any post-regression work via merge rather than overwriting, (c) proposing the proper-shape fix even if it's bigger than the band-aid. They're willing to take the slower-but-correct path for guest-facing content.

### `npm run prebuild` quirk: sync-sot reverts vendored-only fixes locally

When `_shared/properties/` is locally accessible (Mac mini), `npm run prebuild` runs `sync-sot.mjs` which copies upstream over the vendored `data/sot/properties/`. Any vendored-only changes (e.g. PR #12's `usage_sections` removal) get reverted locally on every build. Vercel CI doesn't have `_shared/`, so it uses the committed copy — production is unaffected.

Workaround during the QI PR: `git checkout main -- data/sot/properties/` to discard sync-sot's reversions before committing.

Worth a backlog item: make sync-sot a non-destructive merge or skip when vendored copy is newer than upstream. Avoids the local footgun.

### Translator now MDX-aware

`scripts/translate-property.mjs` gained a third `kind` (`mdx`) alongside `yaml` and `markdown`. The MDX prompt rules:
- Frontmatter: translate only `title:` and `description:`; keep other keys verbatim.
- Component invocations: keep verbatim; **add `lang="zh"`** to every component without one.
- Markdown body: translate prose; preserve all formatting.
- Brand names (Bosch, Daikin, Sonos, etc.): verbatim per existing rules.

Used to translate ~30 EN MDX files into ZH this session at ~$0.10 each.

### Locale-tolerant middleware (PR #8, refined PR #13)

`src/middleware.ts` now has `decomposeDocsPath()` which strips an optional `/<lang>/` segment from the URL. All cookie-gate matchers (slug regex, blocked prefixes, Short.io handshake regex) run against the locale-stripped path. Adding a locale = adding it to the `LOCALES` set, no other middleware changes.

Also sets `x-pathname` header on every response so `RootLayout` can read it via `headers()` and emit `<html lang="zh-Hans">` on ZH routes vs `<html lang="en">` elsewhere. Marketing routes default to `en` if the header isn't set.

### `go.snowbirdhq.com` short links — completely unaffected

Verified during PR #8 planning. Short-link redirector (`src/app/s/[slug]/route.ts`) and the SHORT_LINKS map (50+ entries) are locale-agnostic. All entries point to `/docs/properties/<slug>[/...]` on the docs subdomain. Phase B route restructure doesn't touch the host-conditioned `go.*` rewrites in `next.config.mjs`. EN URLs unchanged → existing short links resolve identically pre/post.

### Tooling built today

- `scripts/scaffold-zh-property.mjs` — generates 4 ZH MDX files for a property from the EN MDX templates. Idempotent. `--dry-run` flag.
- `scripts/list-zh-properties.mjs` — scans `content/docs/zh/properties/` and emits `src/lib/zh-properties.ts` with the Set of translated slugs. Wired into prebuild + scaffold so `LocaleSwitcher` never has a stale pilot list.
- `scripts/zh-batch.sh` — serial wrapper around translator + scaffold for many slugs at once. Used to roll out 12 new properties in ~12 minutes wall-clock.
- `scripts/check-translation-freshness.mjs` — already existed; gained more manifest coverage (18 manifests total now).
- `scripts/empty-usage-sections.mjs` — one-shot for removing redundant YAML data after MDX restoration. Kept as documentation.
