### LocaleSwitcher positioning — four iterations to find the right slot

1. **Floating `top-3 right-4 fixed`** (inherited from ZH pilot) — overlapped fumadocs hamburger on mobile.
2. **Floating `right-16 lg:right-4`** — still overlapped because fumadocs Nav has its own sticky stacking.
3. **`nav.children` slot with `ml-auto`** — moved out of fixed positioning into the nav header, but landed in fumadocs' `<div flex flex-1>` container DOM-adjacent to the hamburger; `ml-auto` pushed the chip right against it. Visually still "floating on top of" the hamburger.
4. **`nav.title` slot wrapping logo + chip in a flex container** — Title renders FIRST on mobile Navbar (before search and hamburger). Three distinct elements, no right-edge competition. Right answer.

Fumadocs nav-slot rendering insight saved to global MEMORY.md (general knowledge for any future fumadocs project).

### Compact `EN ▾` dropdown shrinks footprint ~55%

User feedback after the nav.title fix: chip still dominated header real estate at ~120px wide (3-pill `EN · 中文 · 日本語` row). Industry research surfaced four common patterns: globe icon + popover (Apple), compact code + chevron (Stripe), plain text row, footer-only (Notion). User picked Stripe-style. Final shape: `EN ▾` / `中文 ▾` / `日本語 ▾` showing only the current locale, click opens a 3-item dropdown with check-mark beside current. ARIA complete (`aria-haspopup`, `aria-expanded`, `role="menu" / "menuitem"`, `aria-current`), closes on outside-click + Escape, chevron rotates 180°.

### `<PropertyUsageSections>` → MDX migration gotchas

1. **MDX autolink `<https://url>` breaks Turbopack parser.** Markdown's autolink shorthand renders as a JSX tag start in MDX — `<` followed by `h` then a slash later in `https://` looks like an unclosed JSX tag. Convert to bare URL or `[url](url)` form. Saved in MEMORY.md.

2. **Strip-regex `(?=^###?\s|\Z)` greedy-matches to EOF** when stripping the LAST H3 section, eating any non-heading content that follows (in this case `<ApplianceSet />`). Future migrations need anchored termination — match until next `<H2>`, blank-line + component tag, or explicit terminator. Not yet folded into `migrate-usage-sections-to-mdx.py`.

3. **Pre-commit `end-of-file-fixer` runs AFTER staging**, modifying file content/mtime. Manifest sourceHash computed before staging is now stale → Vercel CI freshness check fails. Workaround: compute hashes AFTER all formatters run, or refresh in a follow-up commit. Saved in MEMORY.md.

### Appliance library architecture decision

User flagged "Should the top section be general principles to heating, kitchen etc, then the appliances section should be specific on how to use each appliance?" — clean architectural insight. Implemented:

- **Inline MDX** kept Property Orientation only: Internet / Heating Guide (concept-level, "slab system, takes 2-4 hours to adjust") / Rubbish & Recycling / Children's Amenities.
- **`_appliances/<slug>.md` library** holds all device-operating instructions with a `category:` frontmatter field (kitchen / heating / climate / laundry / wellness / tech / outdoor / smart-home).
- **`<ApplianceSet>` groups by category** under the `## Appliances` H2, auto-shifts each appliance body's headings H3 → H4 inside the group.

Property-specific entries (gas fire, wood burner, spa, electric blinds, electric windows) use slug-namespaced names (`25-dublin-gas-fire`); model-based entries (Bosch oven, Tefal iron, Sonos) use the model identifier. Soft convention; library mechanically supports both.

Rejected alternative: keep all operating instructions inline. Rejected because it makes the page hard to scan ("where are the appliances I need to operate?" vs "how does heating work conceptually?") and prevents cross-property reuse of the same model.
