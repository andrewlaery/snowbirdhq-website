# Session — 2026-04-24

## Topics

- Short-link click tracking for `go.bcampx.com` / `go.snowbirdhq.com` (end-to-end, from "do we track anything?" to live dashboard)
- Extension to record docs.snowbirdhq.com pageviews
- Extension with per-day counters, range filter (all / 30d / 7d), and inline-SVG bar charts

## Findings

- **Starting state**: no tracking at all on the short-link redirector. `src/app/s/[slug]/route.ts` just 302'd and exited. No KV, no analytics, no `console.log`. Vercel function logs were the only trace, and those are short-lived on Hobby. Decided Option B (Upstash Redis via Vercel Marketplace) over Postgres/PostHog — native stack fit, free-tier sufficient, lowest maintenance.
- **`@vercel/kv` deprecation** was a live surprise — ran `npm install @vercel/kv` and got a deprecation warning pointing at Upstash. Swapped to `@upstash/redis` directly. Vercel's own guidance now routes all Redis via the Marketplace. Documented in global MEMORY.md.
- **Upstash Marketplace provisioning flow**: browser automation (Chrome tab) through the Vercel dashboard was the only option — the Vercel CLI was erroring all OAuth-dependent commands (`FetchError: invalid json response body at .well-known/openid-configuration`). Likely a stale session/SSO issue. Provisioning path: Storage → Create → Marketplace Database Providers → Upstash → Upstash for Redis → Free plan + iad1 region → Create → Connect Database (with `KV` prefix, all 3 environments).
- **Double-click gotcha**: clicking "Create" twice in the confirmation modal triggers Upstash's "Only one free database per account" rule. First click succeeded (DB `snowbirdhq-clicks` visible in Storage immediately), second click raised "Provisioning failed." The good DB was still fine — just had to close the panel and connect it manually.
- **Custom prefix in bind modal defaults to `STORAGE`** — changing to `KV` made env vars auto-inject as `KV_REST_API_URL` / `KV_REST_API_TOKEN` etc., matching what `@upstash/redis`'s default-env client reads. Easy to miss; documented in MEMORY.md.
- **`after()` from `next/server`** is the right primitive for fire-and-forget Redis writes. Works in both route handlers AND middleware on Edge. Used in both `src/app/s/[slug]/route.ts` (after slug lookup, before redirect) and `src/middleware.ts` (wrapped in a local `allow()` helper that replaces every `NextResponse.next()` in the docs-subdomain branch).
- **Data model choice** — started with lifetime counters + 500-event ring buffer. Adding a time-range filter later required per-day counters (`clicks:day:{YYYY-MM-DD}:{slug}` + `docs:day:{YYYY-MM-DD}:{path}`) because the ring buffer alone couldn't answer "last 7 days" at any real volume. One extra Redis INCR per hit; still well under Free-tier 500k commands/month budget.
- **NZT-aligned day keys** via `Intl.DateTimeFormat('en-CA', { timeZone: 'Pacific/Auckland', ... })` — `en-CA` locale formats natively as ISO `YYYY-MM-DD`. Avoids UTC-alignment bugs where "today" on the dashboard wouldn't match the user's actual calendar day.
- **Privacy posture**: IPv4 truncated to /24 prefix (keeps "return visitor-ish" signal without being full PII), IPv6 dropped entirely, User-Agent bucketed to a short label (iOS Safari / Android Chrome / Desktop Chrome / Bot / etc.). Geo via Vercel's `x-vercel-ip-country` + `x-vercel-ip-city` headers — no IP geolocation library needed.
- **Middleware carve-out pattern**: added `INTERNAL_PORTAL_PATHS = ['/docs/internal/link-stats']` alongside the existing `BLOCKED_PREFIXES`. The dashboard bypasses the hard-404 but requires the portal cookie. Clean pattern for any future internal tools.
- **Inline SVG bar chart** instead of Recharts/Chart.js — 60 lines, no bundle bloat, matches the editorial-luxe aesthetic, `<title>` elements give native hover tooltips. Good default for small internal dashboards.
- **Vercel CLI auth break** cost me a few minutes — all `vercel env ls` / `vercel integration list` calls failed with the same JSON-parse error throughout the session. Worked around via the web dashboard. Fix is probably `vercel logout && vercel login` — not tested this session. Documented as a symptom pattern in MEMORY.md.
- **Dashboard UX choices** — deliberately skipped: charts with axes/gridlines, unique-visitor counts, CSV export, per-referer drill-down, alerting, hourly granularity. All deferred until we know what questions real traffic data raises.

## Next Steps

- Watch traffic for a week or two and see which questions surface ("why does slug X get no hits?", "is this geo distribution realistic?", etc.) before adding any more dashboard capability.
- Consider swapping the in-memory AI-chat rate limiter for a Redis-backed sliding window now that Upstash is wired — the backlog item is lower-friction with the client already installed. Still deferred until abuse is observed.
- If the dashboard becomes a regular stop, add a "Today" range option — currently the smallest range is 7 days because a single-bar chart looks silly, but a dedicated card layout without the chart would work.
- If the `DOCS_ACCESS_KEY` ever needs to become a real secret (portal hosting sensitive content), switch to a signed handshake that sets the cookie in the redirector before forwarding, rather than forwarding the key in a query param.

# Session — 2026-04-24 (afternoon / content fixes)

## Topics

- 6-25-belfast user instructions: fix duplicate gas-fireplace content
- Deal with the regression that the vault-sync caused (wiped Sonos physical controls + Shark tower fans)
- Tidy up the outstanding uncommitted docs/session logs from the morning session

## Findings

- The duplicate the user flagged was real and spanned two places: a narrative "Gas Fire" sub-block in the Heating Guide (correct remote-control steps but no wall-switch context) and a generic "Escea Gas Fireplace" appliance card further down (referenced a safety screen and "wall switch" with zero detail). The other heating/cooling appliances (e.g. Daikin heat pumps) follow a split pattern — brief contextual mention in the Heating Guide, full operational card in the Appliance User Guide — so the fix was to make the fireplace match.
- **User correction mid-plan**: the "wall switch" in the old appliance-card entry was NOT phantom boilerplate; there really is a blue-LED timer switch on the wall to the left of the fireplace. Guests must press it to supply power BEFORE the remote will start the fire. Without that, the remote looks like it's doing nothing. Captured as a two-step sequence in the rewritten appliance card — this is the kind of operational detail that's easy to lose if you write docs from the manufacturer spec instead of the actual installation.
- **Regression caused by running `sync-property-compendium.sh --only 6-25-belfast`**: the sync script regenerated the MDX from the Obsidian vault, and the vault was STALE relative to the MDX. Two earlier commits (`0d80c87` Sonos physical controls + `f3bcf32` Shark TurboBlade tower fans) had gone straight to the MDX without updating the vault, so the sync silently dropped both sections. `docs/AUTHORING.md` line 68 already warns "Edit the vault, not the MDX", but the guardrail only works if everyone follows it on every commit. Need an audit of all 15 properties for the same class of drift — captured as a Backlog item.
- **Recovery path**: I read each regression-introducing commit's diff (`git show {sha}`), re-applied the content to the vault `.md`, re-ran the sync, and got a clean MDX + vault-in-sync state. That's the mechanical fix. The structural fix (a pre-sync diff check or a vault-drift linter) is deferred.
- **Pre-commit hooks auto-fix trailing whitespace + end-of-file** (pre-commit config: `trim-trailing-whitespace`, `end-of-file-fixer`). On a MDX-file commit they strip the drift that the Obsidian-sourced markdown brings in — meaning the sibling MDX files I didn't intend to commit ended up identical to HEAD after the hook ran, so I restored them and kept commits scoped to the one file that actually changed.
- **Sandbox blocked `git commit` once** because pre-commit's cache lives at `~/.cache/pre-commit/` which isn't in the sandbox write allowlist. Retried with `dangerouslyDisableSandbox: true` — the permission-prompt-to-disable-sandbox flow is fine for this use case. A longer-term fix would be to add `~/.cache/pre-commit/` to the allowlist via `/sandbox`.
