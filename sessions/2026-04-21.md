### Architecture decisions

**Self-host on existing Vercel vs bcampx Docker vs Shlink vs keep Short.io**: chose Vercel because (a) already on Pro plan so incremental cost is zero, (b) destinations live in git (reviewable, reversible, no dashboard), (c) no API keys to rotate, (d) "if the website is up, short links are up" — one fewer thing to monitor.

**Three-kind redirect schema** (`property` / `path` / `external`):
- `property` handles the bulk (slug + optional page, auto-appends `?access=<DOCS_ACCESS_KEY>`)
- `path` covers shared pages like Queenstown Insights without special-casing
- `external` covers the one non-docs link (34 Shotover car-park Google Drive)
- Single schema, simple lookup, typed const map. No DB, no dashboard.

**Route implementation**: Edge runtime, GET only, unknown slug → 404 (not a default landing — deliberate, no path-enumeration signal). Access key read from env at request time so rotation doesn't require touching the map.

## Findings

### TCC blocker resolution
- Write tool's "must Read first" guard: successful Read only. EPERM errors don't count — guard refuses to let Write overwrite an existing file that TCC has blocked from being Read. However, Write CAN create brand-new files in the same TCC-protected directory (confirmed by creating `_test_write.md` inside `6-25-Belfast/`). So the syscall path Write uses isn't blocked — only the Read syscall is.
- Bash TCC restrictions persist even with `dangerouslyDisableSandbox: true` — that flag bypasses Claude Code's own sandbox but not the OS-level TCC layer.
- **Fix for future sessions**: System Settings → Privacy & Security → Full Disk Access → `+` → `/Applications/iTerm.app`, toggle ON, **⌘Q iTerm completely and relaunch**. Then `cat ~/Documents/...` works. One-time user-side action; no CLI or tool config change needed.
- Saved to project memory (`reference_tcc_iterm.md`) and global MEMORY.md under the sandbox/tool-behaviour cluster.

## Findings

### property_profile_store — the canonical per-property tracker
`__server-bcampx/guest-ops/src/property_profile_store.py` holds the SQLite `property_items` table. Schema: `id` (8-char uuid), `listing_id`, `property_name`, `category` (hazard/maintenance/amenity/compliance/process/feedback_theme/improvement), `title`, `description`, `status` (open/in_progress/resolved/dismissed/monitoring), `priority` (low/medium/high/critical), `resolution_note`, `resolved_at`, `created_at`, `updated_at`. Also: `property_item_evidence` linking table with unique constraint on `(item_id, source_table, source_id)`. Lifecycle API: `add_item()`, `update_item()`, `update_status()`, `add_evidence()`, `get_items()` with filter params, `get_property_summary()`, `get_portfolio_summary()`.

Property name convention: **`internal_name` from `snowbirdhq.properties.PROPERTIES`** (PascalCase: `6-25-Belfast`), NOT the web slug (lowercase: `6-25-belfast`). `listing_id` is the Hostaway listing ID as a string.

### PROPERTIES registry shape (discovered while probing)
Turns out `PROPERTIES` is a `dict` keyed by Hostaway listing_id (string). Values are dicts with `internal_name`, `address`, `nicknames` (list), `commission`, `owner_name`, `owner_email`, `xero_tracking`, `bookingcom_property_id`, `ota_accounts` (airbnb/bookingcom/vrbo), `amenities` (property_type, has_spa, has_garage, garage_type, has_cameras, camera_locations, has_noise_monitor, max_adults). 14 entries total. ### property_items has a dedup gap
`add_item()` has no uniqueness check on `(property_name, title)`. The evidence table has `UNIQUE(item_id, source_table, source_id)` but items themselves are free to duplicate. `seed_property_profile.py` relies on the evidence uniqueness constraint to skip already-imported rows, but that only prevents double-linking of the same source — it doesn't prevent multiple items for the same underlying issue. Today's dedup pass found 3 duplicates all created 2026-03-01 (SkyCrest sauna light + 73B-Hensman washing machine + 73B-Hensman hot water). Worth a tightening pass on `add_item()` or `seed_property_profile.py`.

My first recommendation. User correctly pushed back. `ha-controller` tracks HA-monitoring-specific issues (device_offline, automation_failure, temperature_issue, security_event, energy_anomaly, connectivity) with a `manual` escape hatch. Keyed by `property_id` (UUID-ish) + `entity_id`. Designed for the HA agent loop, not general property operations. property_profile_store in guest-ops is the purpose-built home.
