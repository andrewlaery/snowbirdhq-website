## Topics

- Built and shipped a Simplified Chinese pilot for `/properties/7-suburb` end-to-end (landing + 4 sub-pages + AMA chat)

## Findings

### Architecture decision: narrow `/zh/*` route, NOT full Fumadocs i18n

Three options were considered before committing:
1. **Full Fumadocs i18n** — restructure every existing route to add `[lang]` segment, add `i18n` config to `source.config.ts` and `loader()`, fork the content tree. Framework-blessed but high blast radius for a 1-property test (12 other property pages need to be re-routed).
2. **Narrow pilot route** — add `src/app/zh/properties/[slug]/...` only, leave existing routes untouched. Lower blast radius, doesn't scale gracefully but is deletable.
3. **In-place locale via slug-prefix detection in `[...slug]` catch-all** — peek the first segment, strip if `zh`, load `.zh.mdx`. Fights the framework, brittle.

Picked (2). When a 2nd property/locale arrives, promote to (1) using the same translation pipeline + `lang`-aware SOT loader unchanged.

### Translation pipeline: `scripts/translate-property.mjs`

- Sonnet 4.6 (`claude-sonnet-4-6`), max 16k output tokens, NZ-aware prompt with explicit place-name list (皇后镇 = Queenstown, 新西兰 = New Zealand, 弗兰克顿 = Frankton; keep street addresses + phone numbers + access codes English).
- SHA-256 of source content stored in `data/sot/properties/<slug>/zh/.translation-manifest.json`. Re-runs skip files where source hasn't changed.
- Actual cost on first run: ~14k input + 19k output Sonnet 4.6 tokens ≈ **$0.36** for 7-suburb (38k chars `guest_copy.md` + 9k `facts.yaml` + 1k `identity.yaml`).

### Gotcha: Fumadocs `nav.children` doesn't lay out as expected

Adding `<LocaleSwitcher />` to the EN docs layout via `nav={{ title, children: <LocaleSwitcher /> }}` rendered the switcher jammed against the "DOCS" badge with the Chinese characters wrapping vertically (中 stacked over 文). `ml-auto` had no effect. Fix: remove from `nav.children`, drop a fixed-position pill outside `DocsLayout` instead. Memory entry created at `~/.claude/.../memory/fumadocs_nav_children.md` for future sessions.

### Bug discovered while wiring the ZH pilot: `PropertyWelcome` is silently broken on EN

`src/components/property-welcome.tsx` calls `prose.sections['Welcome']`, but `extractSections` in `src/lib/sot.ts` only matches `^## ` (H2). The `guest_copy.md` files use `# Welcome` (H1). So the section lookup returns `undefined` and the component returns null on every property page. Verified by reading the EN file structure during the ZH wiring. Two fixes possible (extend extractor to H1, or change source files to H2). Logged as a backlog item — not fixed in this session.

### AMA architecture: context stays English, output in Chinese

Considered translating the SOT context too so the AMA reads ZH and answers ZH. Rejected because: (a) it doubles cache footprint with two locale-tenant prompts, (b) creates an EN/ZH drift risk on the source of truth, (c) Haiku 4.5 is fluent enough to read English context and reply in Chinese natively. Confirmed live: `WiFi 密码是什么？` → returned `Wright's Garden 的 Wi-Fi 密码是 ThankYou4Visiting, Wi-Fi 网络名称是 WrightsGarden, 祝您上网愉快！` with credentials lifted correctly from English `facts.yaml`.

### Pre-commit hook: gitleaks false-positive on translated WiFi password

The same WiFi password (`ThankYou4Visiting`) lives in the original EN `facts.yaml` already committed to the repo, but gitleaks only scans new content — the ZH file got flagged as "generic-api-key" (entropy 3.69). Resolved by inline `# gitleaks:allow guest-facing wifi shown publicly on docs portal` rather than allowlisting in `.gitleaks.toml` — matches the redact-by-reference pattern in MEMORY.md gitleaks-false-positives entry.
