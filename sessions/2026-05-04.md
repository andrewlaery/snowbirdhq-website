### The audit caught much more than the immediate question

The user asked "is anything else missing on SkyCrest?" and the answer turned out to apply portfolio-wide. The migration treated `content/docs/properties/<slug>/user-instructions.mdx` as ground truth, but the docs MDX was downstream of and older than the upstream SOT in many cases. This blind spot meant we inherited every gap that already existed in the docs.

Audit method (saved as `scripts/verify-portfolio-rollout.py` for future reuse):
1. **Token-set coverage**: pre-migration MDX → post-migration MDX+library. Catches "did I lose anything from the file I migrated FROM" but NOT "is the file I migrated FROM complete".
2. **5-file SOT cross-check** (delegated to 3 parallel research agents): for each property, read `facts.yaml::usage_sections`, `policies.yaml::known_issues`, `owner_log.md`, `ota_copy.md`, `AUDIT.md` against the migrated docs+library. This is what caught the SkyCrest gaps and 10 other properties' issues.

## Findings

### `<HouseRulesBase slug="..." />` slug-templating

The component now accepts an optional `slug` prop. When provided, `{slug}` template tokens in `data/sot/_shared/house-rules-base.md` get substituted at render time. Enables per-property links in shared markdown for the universal "Appliance Use → [User Instructions](/docs/properties/{slug}/user-instructions)" and "In an Emergency → [Critical Information](/docs/properties/{slug}/critical-info)" cross-links. Falls back gracefully (token preserved) if no slug passed.

### Composition pattern — final shape (all 14 properties)

```
welcome-house-rules.mdx (~17 lines):
  <PropertyQuickInfo slug="..." />
  <PropertyWelcome slug="..." />
  <HouseRulesBase slug="..." />
  ## Property-Specific Rules
  <PropertyHouseRulesDeltas slug="..." />
  <QueenstownEssentials />

critical-info.mdx (~10 lines):
  <CriticalInfoBase />
  <PropertyOperationalNotes slug="..." />
  <PropertyHazards slug="..." />
```

Total: 168 MDX files (14 properties × 4 page types × 3 locales). Universal changes to shared components ripple to every property automatically.

### Three near-disasters

1. **My "all 14 on composition" commit (8f27c8c) only included SOT changes, not MDX.** The agent runs (A + B + C) wrote composition shells locally, but pre-commit hook interactions, partial staging, and rebase-on-push collisions resulted in the MDX changes getting lost between agent completion and commit. Discovered because the user asked "are 10b-DeLaMare changes live?" — auth-gated curl confirmed 7 of 14 properties were still rendering legacy 100-160-line inline MDX. Commit `2a58c54` actually fixed this with proper composition shells (-4780 lines deleted).

2. **Curl verification without auth was meaningless** — first attempt to verify production showed all markers as "0 hits". Looked like nothing had deployed. Actually because the docs portal is auth-gated (`docs_portal` HMAC cookie required). Curl was hitting the access redirect page, not the property page. Workaround:

   ```bash
   curl -sS -i -X POST https://docs.snowbirdhq.com/api/access-unlock \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "password=SnowbirdHQ"
   # Capture Set-Cookie: docs_portal=1.<hmac>...
   curl -sL -H "Cookie: docs_portal=1.<hmac>" <gated-url>
   ```

3. **Multi-agent migrations don't reliably complete** — agent A reported "Build clean", agent C hung at 504 tool uses (~74min) but DID complete its work, agent B was killed mid-execution. None of these "completion" reports translated to "actual MDX shells in main". Lesson: after multi-agent migration runs, always verify END STATE structurally (file-by-file check) before claiming done. Don't trust agent self-reports.

### TypeScript target lib doesn't include `String.replaceAll()`

Hit during HouseRulesBase slug-templating. Use `body.split(token).join(value)` instead. ES2021 target would fix but config change is bigger than the workaround.

### Pre-commit hook re-staging pattern (already in MEMORY.md but reinforced)

Every multi-file commit this session followed this pattern:
1. `git add -A`
2. `git commit -m "..."` → hook fixes file (end-of-file-fixer or similar) → de-stages it → "Restored changes from .../patch1777..."
3. `git add -A` again
4. `git commit -m "..."` → succeeds

Sometimes also needs `git restore --staged CHANGELOG.md .claude/scheduled_tasks.lock` to keep unrelated files out of the commit.

### Auth gate cookie shape

`docs_portal` cookie value: `1.<hmac>` where the digit `1` is the version + `.` + 64-char hex HMAC. `Path=/`, `Expires=+1y`, `Secure`, `HttpOnly`, `SameSite=lax`. Set by POST /api/access-unlock with form-encoded `password=<DOCS_PORTAL_PASSWORD>` (currently `SnowbirdHQ`).

### Sandbox quirks during git operations

- **Pre-commit hook** (`/Users/andrewlaery/.cache/pre-commit/`) — `pre-commit` writes a stash patch under `~/.cache/pre-commit/`. Sandbox writes are confined to `/Users/andrewlaery/.local/share/`, `~/.npm/`, `~/.omp/`, `/Volumes/NextCloud/`, project dir, `$TMPDIR`. `~/.cache/` isn't in the allowlist, so any commit that triggers pre-commit fails with `PermissionError: [Errno 1] Operation not permitted` on the patch file. Workaround: `dangerouslyDisableSandbox: true` for the commit. Same will apply on every project that has pre-commit hooks installed.
- **`git push`** (`/Users/andrewlaery/.ssh/known_hosts`) — `~/.ssh` is in the read-deny list. SSH push fails with `hostkeys_find_by_key_hostfile: hostkeys_foreach failed`. Workaround: `dangerouslyDisableSandbox: true` for the push.
