### Mattermost API permission boundaries hit
- Bot token `MATTERMOST_BOT_TOKEN` from guest-ops cannot hit `/api/v4/hooks/incoming/{hook_id}` — returns error (insufficient permissions for webhook metadata)
- Bot token CAN hit `/api/v4/users/me/teams/{team_id}/posts/search` but only returns posts in channels it's a member of — incoming webhooks post without requiring bot membership, so the bot's own post isn't discoverable by search
- Net: cannot programmatically identify "which channel did this webhook post to" without either adding the bot to the target channel or using a system-admin user token
- Resolution used: Andrew looked at the Integrations UI manually

### property_items dedup gap
`PropertyProfileStore.add_item()` has no title-level uniqueness check on `(property_name, title)`. `seed_property_profile.py` relies on the evidence table's `UNIQUE(item_id, source_table, source_id)` constraint to skip already-imported evidence links, but that only prevents double-linking the same source — not creation of parallel items for the same underlying issue.

Portfolio-wide dedup pass today flagged 3 exact-match duplicates (all dismissed) + 1 near-duplicate at 10B-DeLaMare needing judgement (`5db62932` vs `687c941d`, different word order for the same heater/AC issue).

The CLI's `add` command adds an idempotency guard at the client level (refuses duplicate open `(property, title)`) but the underlying API still accepts duplicates via other paths.

### Cron-without-container-restart pattern
Passing env vars via `docker exec -e VAR=VAL` only affects the single exec invocation — not the running main process, not other execs. Useful for cron lines where you want an override without touching `.env` or restarting. Pattern used for `MATTERMOST_PROPERTY_DIGEST_WEBHOOK_URL` routing.

## Findings

### isort / ruff strips function-local imports at module level
Re-hit `til-2026-04-21-isort-strips-function-local-imports` today. Initial edit added `import socket` at module level but only used `socket.gethostname()` inside `build_digest()`. The pre-commit hook's ruff step stripped the import as "unused" because the static analysis can't see intra-function usage. Fix: move the import function-local (inside `build_digest()`). MEMORY.md has the canonical note — this session just confirmed the pattern bites the same way in this codebase. Added a function-local import with a `# function-local: isort otherwise strips the module-level import` comment as visual reminder.

### Docker `socket.gethostname()` returns container hash, not service name
Inside the `guest-ops` container, `socket.gethostname()` returns the Docker-assigned 12-char hex hash (`f0b85751202b`) rather than the docker-compose `container_name: guest-ops`. First version of the breadcrumb footer surfaced this hash in every post, which is useless to a human reader. Fixed by hardcoding `Container: \`guest-ops\`` as a static string — the value is compose-managed and changes only if someone renames the container, which is rare and intentional.

Also worth noting: `/etc/hostname` inside the container has the same hash. The service name isn't exposed to the container at runtime unless explicitly plumbed via env var (e.g. `- HOSTNAME_LABEL=guest-ops` in compose). Not worth the ceremony for a footer string.

### Git SHA availability inside container
`_git_sha()` does a best-effort `git -C <path> rev-parse --short HEAD` at two possible paths (`/app` and `/home/bcampx/projects/guest-ops`). Neither has a `.git` directory in the `guest-ops` container's filesystem, so the SHA is silently omitted from the footer and the rest still renders. If SHA visibility matters later, easiest path: pass `DIGEST_GIT_SHA=$(git -C /home/bcampx/projects/guest-ops rev-parse --short HEAD)` via `docker exec -e` from the cron line, and have `_git_sha()` read that env var first.

### Shell alias conflict with 1Password CLI
`op` is also the 1Password CLI's canonical binary name. Andrew doesn't currently have it installed (`which op` → not found), but if they do later, this alias will shadow it. Flagged in the end-of-session note. Safe alternatives: `opp`, `ops`, `prop`, `bcampx`.
