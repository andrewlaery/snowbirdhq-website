## Findings

### Fumadocs `valueToExport` gotcha
`remarkStructure` is enabled by default in fumadocs-mdx 14.x (`exportAs: "structuredData"`), but the `valueToExport` array in `applyMdxPreset()` controls which `file.data.*` keys get emitted as module exports. Default is `[]`. Result: `structuredData` gets computed but never exported → `page.data.structuredData` is `undefined` at runtime → advanced search throws `Cannot read properties of undefined (reading 'headings')`. Fix: `defineConfig({ mdxOptions: { valueToExport: ['structuredData'] } })`. Run `npx fumadocs-mdx` to regenerate `.source/` after the config change.

### Fumadocs canonical search wiring
`createFromSource(source)` from `fumadocs-core/search/server` is the single-line replacement for the manual `createSearchAPI('advanced', { indexes: ... })` boilerplate. Takes the source directly, handles `structuredData` extraction, generates `id` from `url`. Simpler + correct by default.

### Vercel Deployment Protection strips the original path
Preview URLs are gated by Vercel SSO. User visits `*.vercel.app/docs` → 401 → SSO challenge → after auth, redirects to `*.vercel.app/` (root), NOT back to `/docs`. Any docs flow that assumes "user hits /docs and our middleware runs" breaks on preview. No Vercel-side fix without enabling "Protection Bypass for Automation" (generates a bypass token). Workaround: tell user to navigate to `/access` directly in the same SSO-authed tab.

### `next-themes` under RootProvider auto-detects `prefers-color-scheme`
Fumadocs `RootProvider.theme` passes through to `next-themes` ThemeProvider. Default is `enableSystem: true` + `defaultTheme: 'system'`, which picks up the OS dark-mode preference. When a design system only defines light tokens, this causes partial cascade failures (sidebar uses `--fd-card` which can be overridden, but body uses `--fd-background` with `.dark` selector outside the override scope). Lock with `forcedTheme: 'light'`.

### Vercel env-var scope: Preview vs Production
Env vars have a three-way scope (Development / Preview / Production). Our `DOCS_*` vars were Production-only — Preview deployments had no secrets, causing the password gate to silently fail. Propagated via `vercel env pull --environment=production` → `printf '%s' $value | vercel env add NAME preview --force`. Always `printf '%s'` to avoid the trailing-`\n` corruption flagged in the global MEMORY. `--force` lets you overwrite existing preview values.

### Fumadocs CSS variables are HSL triplets
Fumadocs uses `hsl(var(--fd-background) / 1)` pattern — variable values are unquoted HSL triplets (`36 35% 95%`), not full `hsl(...)` strings. Conversion from hex required: `#F6F2EC` → `36 35% 95%`. Any override must match this format to cascade correctly.

### Fumadocs sidebar customisation hooks
- `id="nd-sidebar"` is a stable ID for top-level sidebar targeting.
- Active link: `a[data-active="true"]` — stable for CSS styling.
- Folder labels: rendered via `SidebarFolder` + `SidebarFolderTrigger`/`SidebarFolderLink` internally. No stable data attribute on the trigger button — CSS targeting by position is fragile. To restyle folder labels, use `DocsLayout.sidebar.components` prop to override `SidebarComponents.Folder`, but that requires reimplementing Radix Collapsible state (non-trivial). Deferred.
- `disableThemeSwitch` prop on `DocsLayout` removes the sun/moon toggle from UI but doesn't prevent auto-detect — still need `RootProvider.theme` config for that.

### Heading-link underlines look clunky at serif size
Hairline `border-bottom: 1px solid var(--snow-line-2)` works well for body-prose links at 16px sans-serif. At 40-56px Newsreader serif, the hairline stretches the full text width and breaks awkwardly on wrapped lines. Rule: suppress `border-bottom` when the link is inside `h1/h2/h3/h4`. Keeps the colour-hover-shift affordance (text changes to accent), drops the clunky underline.

### `gh pr merge` fast-forwards local main
After `gh pr merge PR#1 --squash --delete-branch`, local HEAD auto-updates to include the squashed commit on main (via a fast-forward pull). Subsequent commits in the same shell session go straight to local main — if you then try to push, you're pushing direct to main, not to a feature branch. Pattern: before committing after a merge, explicitly `git checkout -b feature/xyz` to avoid accidental direct-to-main commits.
